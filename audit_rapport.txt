════════════════════════════════════════════════════════════════════════════════
  1. SERVICE WORKER  (sw.js)
════════════════════════════════════════════════════════════════════════════════

CACHE_NAME : 'invo-v10'
Stratégies :
  • Cache First  → shell + assets statiques (JS, CSS, fonts, images, manifest)
  • Network First sans cache → tout autre chemin (futur API, routes inconnues)
  • Offline Fallback → retour index.html (navigation), JS vide, CSS vide, 503

PRECACHE_ASSETS : 71 fichiers listés explicitement (critiques + optionnels)
  Critiques  : index.html, tous JS, tous CSS, fonts WOFF2, icônes, manifest
  Optionnels : screenshots/ (non bloquants si absents)

CYCLE DE VIE :
  install  → précache critique, skipWaiting auto en cas de succès
  activate → purge tous les anciens caches (≠ CACHE_NAME), clients.claim()
  fetch    → routing par URL, ignore chrome-extension://
  message  → SKIP_WAITING accepté depuis sw-register.js

POINTS POSITIFS :
  ✅ Stratégie Cache First cohérente pour assets statiques
  ✅ Purge automatique des anciens caches à l'activation
  ✅ clients.claim() → prise de contrôle immédiate sans rechargement
  ✅ Offline fallback pour HTML, JS, CSS
  ✅ Broadcast postMessage vers les clients en cas d'échec critique
  ✅ Ressources optionnelles (screenshots) non bloquantes
  ✅ Ignorer chrome-extension:// (bonne pratique)

PROBLÈMES DÉTECTÉS :
  [P1-MINEUR] sw.js lui-même est dans PRECACHE_ASSETS. Si le SW change, le
              navigateur le re-télécharge de toute façon (double effort, pas
              de bug réel, mais c'est inutile de le précacher).

  [P2-MINEUR] isAppShellRequest() inclut tout .js/.css/.png/.svg/.woff2.
              Si un futur endpoint API retourne .js (ex. /api/data.json),
              il serait mis en Cache First par erreur. Non problématique
              aujourd'hui car 100 % local, mais fragile à l'évolution.

  [P3-MINEUR] networkFirstNoCache ignore silencieusement tous les échecs
              fetch (catch _). Aucun log de débogage même si DEBUG=false.
              Si une ressource dynamique échoue hors-ligne, l'utilisateur
              voit juste un 503 sans message d'erreur.

  [P4-INFO]  CACHE_NAME = 'invo-v10'. La checklist demande d'incrémenter
             avant chaque déploiement modifiant des assets. Risque d'oubli
             si pas automatisé.

  3. STOCKAGE LOCAL (OPFS + localStorage + sessionStorage)
════════════════════════════════════════════════════════════════════════════════

Fichier principal : /js/storage.js (896 lignes)

Couches de stockage :
  • OPFS (Origin Private File System) : couche principale
    - Fichiers JSON versionnés : enveloppe { STORAGE_FORMAT_MARKER, data, ... }
    - STORAGE_FORMAT_VERSION = 2
  • localStorage : fallback (anciens navigateurs) + migration auto
  • sessionStorage : session lock uniquement (invoo_app_session_lock)

Clés OPFS :
  invoo_settings, invoo_clients, invoo_stock, invoo_docs,
  invoo_fournisseurs, invoo_bons_commande, invoo_stock_moves
  invoo_activation_v3 (auth), invoo_db_version

Clés localStorage :
  invoo_device_seed_v1 (device fingerprint)
  invoo_ios_hint (timestamp dernier hint iOS)

DB_VERSION : 1 (migrations documentées requises si incrément)

Chaîne d'écriture : _opfsWriteChain (Promise.resolve()) — évite les écritures
  concurrentes et les états incohérents. Bonne pratique.

POINTS POSITIFS :
  ✅ OPFS préféré à localStorage (meilleur isolement, pas de limite 5 MB)
  ✅ Fallback localStorage documenté pour compatibilité
  ✅ Migration automatique localStorage → OPFS
  ✅ Versionnement explicite (DB_VERSION, STORAGE_FORMAT_VERSION)
  ✅ Chaîne d'écriture séquentielle pour éviter la corruption
  ✅ Cache mémoire (_opfsMemCache) pour lectures rapides
  ✅ window.APP.opfs encapsulation partielle (get/set via defineProperty)

PROBLÈMES DÉTECTÉS :
  [P9-IMPORTANT] Les données OPFS ne sont PAS chiffrées. Sur un appareil
                 partagé ou volé, toutes les données métier (clients, factures,
                 montants) sont lisibles via les DevTools ou l'API OPFS.
                 Pour usage personnel, risque acceptable ; à documenter.

  [P10-MOYEN] window.DB est un objet global exposé. N'importe quel script
              peut le modifier directement (window.DB.clients = []). L'encap-
              sulation via window.APP.opfs est partielle ; window.DB reste
              non protégé.

  [P11-MINEUR] Pas de schéma formel (TypeScript / JSDoc complet / JSON Schema)
               pour les entités (client, document, produit). Les migrations
               manuelles sont risquées si un champ est oublié.

  [P12-MINEUR] En cas d'échec OPFS ET localStorage, les données sont perdues
               silencieusement. Aucune UI d'avertissement "stockage indisponible"
               n'a été trouvée dans storage.js.

  [P13-INFO]  La clé invoo_device_seed_v1 reste en localStorage (pas migrée
              en OPFS). Si l'utilisateur efface localStorage, le device ID
              change et l'activation est potentiellement perdue.

              Aucun skeleton screen ou spinner de chargement initial visible

               peu utilisés. Économie possible : ~40-60 KB WOFF2.

  [P17-MINEUR] page-templates.js (865 lignes) est un fichier généré par script
               (build-page-templates.mjs). S'il n'est pas régénéré après
               modification d'un template source, les pages affichent du contenu
               obsolète sans avertissement.

  default-src 'self' data: blob:

POINTS POSITIFS :
  ✅ script-src 'self' strict (pas de CDN, pas d'inline JS)
  ✅ frame-ancestors 'none' (protection clickjacking)
  ✅ object-src 'none' (pas de plugins Flash/PDF externes)
  ✅ base-uri 'self' (protection injection base tag)
  ✅ upgrade-insecure-requests + block-all-mixed-content
  ✅ Aucune ressource externe dans la CSP

PROBLÈMES DÉTECTÉS :
  [P18-MINEUR] style-src 'unsafe-inline' autorisé. Nécessaire pour les styles
               inline générés dynamiquement dans sw-register.js et auth.js
               (bannières, toasts). Difficile à éliminer sans refactorer les
               composants inline. Risque faible car pas de données utilisateur
               dans ces styles.

5.2 XSS & Injection HTML
  DOMPurify (purify.min.js) : intégré localement, pas de CDN
  sanitize.js : wrapper DOMPurify.sanitize()
  html-safe.js : helpers escapeHtml(), escapeAttr(), h() (createElement)
  ESLint : règle no-restricted-syntax bloquant innerHTML direct

POINTS POSITIFS :
  ✅ DOMPurify local (pas dépendant d'un CDN)
  ✅ setStaticHtml() obligatoire pour tout innerHTML
  ✅ h() helper createElement sécurisé
  ✅ escapeHtml/escapeAttr disponibles partout
  ✅ highlightSegments() ne crée pas de HTML injectables
  ✅ ESLint bloque innerHTML non documenté

PROBLÈMES DÉTECTÉS :
  Aucun problème XSS critique identifié dans les fichiers analysés.
  Les exceptions innerHTML (html-safe.js, sw-register.js) sont documentées.

5.3 Licence et Authentification (license-activation.js, auth.js)
  Schéma : SHA-256(email | deviceId | SECRET)
  SECRET : 'INVOO3388' (hardcodé dans license-activation.js ligne 7)
  Comparaison : temps constant (XOR octet par octet) — bonne pratique
  Session lock : PBKDF2 100 000 itérations (fort)
  Device ID : seed aléatoire 16 bytes → SHA-256(fingerprint)

PROBLÈMES DÉTECTÉS :
  [P19-IMPORTANT] Le SECRET ('INVOO3388') est hardcodé dans le JS livré au
                  navigateur. N'importe qui avec les DevTools peut le lire
                  et générer des clés valides pour n'importe quel email +
                  device ID. Pour usage personnel, risque acceptable puisque
                  l'app est locale. Si distribuée, ce SECRET est compromis
                  par nature (pas de backend = pas de secret côté serveur).
                  À documenter clairement.

  [P20-MOYEN] L'activation est stockée dans OPFS (invoo_activation_v3) sans
              chiffrement. Sur un appareil partagé, un autre utilisateur avec
              accès aux DevTools peut voir l'état d'activation, le hash du
              mot de passe, et le salt PBKDF2.

  [P21-MINEUR] La session lock (invoo_app_session_lock = '1' en sessionStorage)
               est binaire. Si l'utilisateur n'a pas configuré de mot de passe,
               la session est déverrouillée automatiquement. Aucun avertissement
               ne suggère de définir un mot de passe.
